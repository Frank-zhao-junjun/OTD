### ISSUE-001: `sap-sales-order-cancellation.ts` 单维度阻塞状态分类错误

**位置**: `OTD/src/lib/sap-sales-order-cancellation.ts:152`

**问题**: 当 `blocked.length === 1`（仅一个维度为 D）时，`kind` 被错误地赋值为 `'multi-blocked'`（多维度阻塞），而非 `'single-blocked'` 或 `'process-blocked'`。

```typescript
// 当前代码（第 150-159 行）
if (blocked.length === 1) {
  return {
    kind: 'multi-blocked',        // ❌ 应为 'single-blocked' 或具体维度类型
    headerClosed: true,
    label: `${blocked[0]}已停止`,
    ...
  };
}
```

**影响**: 
- UI 展示的标签与内部 `kind` 不一致（标签说「交货已停止」，但 `kind` 是 `multi-blocked`）
- 下游逻辑若依赖 `kind` 做分支判断（如统计、导出）会产生错误分类
- 与 `OrderCancellationKind` 类型定义中的语义不符

**建议修复**:
```typescript
if (blocked.length === 1) {
  return {
    kind: 'process-blocked',      // 或新增 'single-blocked'
    ...
  };
}
```

**验收**: UAT-SO-016（取消/拒绝状态识别）

### ISSUE-002: `invoiced-7d` 日期范围存在 UTC/本地时区不一致

**位置**: `OTD/src/lib/sap-sales-order-presets.ts:24-38` vs `OTD/src/lib/sap-sales-order-risk.ts:35-65`

**问题**: 
- `last7DaysRange()` 使用 `isoDate(d)` = `d.toISOString().split('T')[0]`，返回 **UTC 日期**
- `requestedDeliveryDueSoonODataRange()` 使用 `startOfLocalDay()` + `formatLocalDateYmd()`，返回 **本地日期**

在中国时区（UTC+8），UTC 日期可能比本地日期 **早 1 天**（例如本地 2026-06-09 02:00 对应 UTC 2026-06-08 18:00）。这意味着：
- `delivery-7d` 按本地日历日计算，边界正确
- `invoiced-7d` 按 UTC 日历日计算，在凌晨时段可能偏差 1 天

**影响**: 
- 跨天场景下（UAT-SO-X06）`invoiced-7d` 窗口可能提前或延后 1 天
- 首页计数与列表结果可能不一致

**建议修复**: 统一 `last7DaysRange` 使用本地日历日逻辑：
```typescript
function last7DaysRange(now = new Date()): { from: string; to: string } {
  const today = startOfLocalDay(now);   // 复用 sap-sales-order-risk.ts 的本地日函数
  const from = addLocalDays(today, -7);
  return { from: formatLocalDateYmd(from), to: formatLocalDateYmd(today) };
}
```

**验收**: UAT-SO-X02、UAT-SO-X06

### ISSUE-003: `invoiced-7d` 实际为 8 天 inclusive 范围

**位置**: `OTD/src/lib/sap-sales-order-presets.ts:34-38`

**问题**: `last7DaysRange` 返回 `{from: today-7, to: today}`，配合 OData `ge` + `le` 运算符后，实际筛选的是 **8 个自然日**（含两端）。

例如今天 6/8，返回 from=6/1, to=6/8，共 8 天。

**影响**: 
- 产品定义「近 7 天已开票」与实际筛选口径不一致
- 与 `delivery-7d`（今日~+7 天 = 8 天 inclusive）口径对称，但命名易误导

**建议**: 
- 方案 A：将 `last7DaysRange` 改为 `from = today-6`（真正 7 天）
- 方案 B：在 UI 摘要和文档中明确说明为「近 8 天 inclusive」或「7 天内+今日」

**验收**: UAT-SO-006、UAT-SO-X02

### ISSUE-005: `delivery-7d` preset 下用户改状态筛选可能产生矛盾条件

**位置**: `OTD/src/lib/sap-sales-order-presets.ts:119-162`

**问题**: 在 `delivery-7d` preset 下，用户若将状态筛选改为「交货 = 完全处理(C)」，`mergePresetWithForm` 会保留 `deliveryStatusNe: 'C'`（硬条件），同时表单产生 `statusField='delivery', statusValue='C'`。

在 `buildSalesOrderListFilter` 中，`deliveryStatusNe` 条件优先于 `statusValue`，导致用户的选择被静默忽略，最终 OData 条件为：
```
OverallDeliveryStatus ne 'C' and OverallDeliveryStatus eq 'C'
```

**实际行为**: `deliveryStatusNe` 先被写入，但 `statusValue` 随后也被写入。在 `buildSalesOrderListFilter` 中：
```typescript
} else if (filters.deliveryStatusNe) {
  parts.push(`OverallDeliveryStatus ne '${odataEscape(filters.deliveryStatusNe)}'`);
} else if (filters.statusValue && filters.statusValue !== 'all') {
```

由于 `deliveryStatusNe` 存在，`statusValue` 分支不会执行，所以最终只有 `ne 'C'`。用户看到 UI 上选了「交货=C」，但查询结果仍是「交货≠C」。

**影响**: UX 困惑，用户可能认为系统有 Bug。

**建议**: 
- 在 `mergePresetWithForm` 中，当 preset 硬条件与用户选择冲突时，禁用该状态维度下拉框，或在 UI 提示「当前快捷视图已固定交货未完成条件」
- 或在 `buildSalesOrderListFilter` 中增加断言/日志，检测矛盾条件

**验收**: UAT-SO-006、UAT-SO-018

### ISSUE-009: 精确订单号查询仍带默认销售组织过滤

**位置**: `OTD/src/lib/sap-sales-order-filters.ts:191-213`

**问题**: 当用户输入精确订单号时，`exactOrderLookup = true` 会跳过 `SalesOrderType` 默认过滤，但 **仍保留** `SalesOrganization`、`DistributionChannel`、`OrganizationDivision` 的默认过滤。

```typescript
if (!exactOrderLookup) {
  parts.push(`SalesOrderType eq '${odataEscape(orderType)}'`);
}
// 以下始终执行，即使 exactOrderLookup=true
parts.push(`SalesOrganization eq '${odataEscape(salesOrg)}'`);
parts.push(`DistributionChannel eq '${SAP_DEFAULTS.distributionChannel}'`);
parts.push(`OrganizationDivision eq '${SAP_DEFAULTS.division}'`);
```

**影响**: 若用户知道订单号但订单在其他销售组织（如 1020），精确查询返回空。

**建议**: 
- 明确产品决策：精确订单号查询是否应突破默认销售范围？
- 若应突破，将 `salesOrg`/`distributionChannel`/`division` 也移入 `!exactOrderLookup` 分支
- 若不应突破，在 UAT-SO-001 中明确说明「精确订单号仍受默认销售范围约束」

**验收**: UAT-SO-001、UAT-SO-004

### OBS-005: `odataEscape` 仅转义单引号

**位置**: `OTD/src/lib/sap-api-client.ts:12-14`

```typescript
export function odataEscape(value: string): string {
  return value.replace(/'/g, "''");
}
```

**评估**: 标准 OData 字符串转义确实只需处理单引号。其他特殊字符（如 `%`、`*`）在 OData 字符串字面量中无需转义。当前实现符合规范，风险低。

### OBS-006: 审计日志写入无并发锁

**位置**: `OTD/src/lib/query-audit-server.ts:19-39`

**问题**: `appendFileSync` 在单进程 Node.js 中是原子的，但在多进程/多实例部署（如 PM2 cluster）下，并发写入可能导致 JSONL 行交错。

**建议**: 当前 MVP 单实例部署无风险。若未来水平扩展，需改用文件锁或集中式日志（如写入数据库/S3）。
